**Binding descriptor design for Elden Ring:**

ER_KEY_INGAME (index 0): This binding holds the actual in-game key that is driven AS AN OUTPUT by the sprint/dodge behavior. It is the outputVk for HoldToToggle (sprint) and EdgeTrigger (dodge). It should not be driven by a behavior of its own — it is a key that the user rebinds (e.g., 'F' for dodge/dash). Give it `BehaviorType::HoldToToggle` with `outputVk = 0` as a sentinel meaning "this binding drives nothing directly — its currentVk is used as the output of other bindings".

Wait — re-examine: In the existing code, `localIngame` is the OUTPUT key (it's what gets pressed/released for sprint and dodge). `localDodge`, `localSprint`, `localTrigger` are the INPUT keys the user holds.

The GenericLogicThreadFn dispatches each binding independently using `binding.currentVk` as the INPUT key and `binding.behavior.outputVk` as the OUTPUT key. This does not naturally capture the Elden Ring "sprint uses the same outputVk as dodge" relationship.

**Resolution (research-recommended approach):** For Elden Ring in Phase 1, use descriptors that encode the relationship directly:

- ER_KEY_INGAME (index 0): Not a behavior binding — it's a configurable key the user rebinds to be the in-game dodge/dash key. Since the generic engine would try to use `currentVk` as its input trigger if given a behavior, give it a no-op descriptor. Use `BehaviorType::HoldToToggle` with `outputVk = 0` — when outputVk is 0, the HoldToToggle handler does nothing. This lets the user rebind this key via the settings UI while the actual sprint/dodge behaviors reference it as their output.

BUT: GenericLogicThreadFn uses `binding.currentVk` as the input polled key, and `behavior.outputVk` as the key to inject. The ER_KEY_INGAME binding's currentVk is 'F' — the engine would poll 'F' as an input and inject outputVk=0 (nothing). That's acceptable as a no-op.

- ER_KEY_SPRINT (index 2): `BehaviorType::HoldToToggle`, `outputVk = 'F'` (the in-game key). BUT the user may have rebound ER_KEY_INGAME to something other than 'F'. In Phase 1 these are hardcoded descriptors, so hardcode 'F'. Phase 2 (JSON profiles) will need to reference `profile->bindings[ER_KEY_INGAME].currentVk` dynamically — that's out of scope for Phase 1.

- ER_KEY_DODGE (index 1): `BehaviorType::EdgeTrigger`, `outputVk = 'F'`, `durationMs = 50`.

- ER_KEY_TRIGGER (index 3): `BehaviorType::LongPress`, `outputVk = VK_ESCAPE` (short-press: fires Escape on tap), `longOutputVk = 'Q'` (long-press: fires Q action), `thresholdMs = 400`.

Note the phase success criterion: "LongPress fires the short-press output on a tap and the long-press output on a sustained hold." The existing Elden Ring TriggerKey was used to fire 'Q' on long hold; on short tap it did nothing. The new behavior: tap Escape key → fires Escape (back/close); hold Escape key ≥ 400ms → fires 'Q'. This matches the phase goal.

**WheelToKey descriptor design:**

The existing TC logic sends `MOUSEEVENTF_WHEEL` with `TC_SCROLL_DELTA_UP = 120` for the scroll-up binding and `TC_SCROLL_DELTA_DOWN = (DWORD)-120` for scroll-down. Map directly to WheelToKey descriptors using the `wheelDelta` field:

- TC_KEY_SCROLL_UP (index 0): `BehaviorType::WheelToKey`, `wheelDelta = 120`
- TC_KEY_SCROLL_DOWN (index 1): `BehaviorType::WheelToKey`, `wheelDelta = (DWORD)-120`

Note: For scroll-down, `(DWORD)-120` is 0xFFFFFF88. This must be stored as DWORD in the descriptor (which it is — `wheelDelta` is DWORD). The GenericLogicThreadFn passes this directly to `inp.mi.mouseData`, which takes DWORD, matching the original TC implementation exactly.
